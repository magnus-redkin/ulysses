# Мозг (Brain) — подсистема мониторинга и защиты от блокировок

## Назначение

Мозг — это управляющая плоскость VPN-сервиса Ulysses Lab. Он не обрабатывает пользовательский трафик,
а следит за состоянием входных шлюзов (щитов), обнаруживает блокировки РКН и управляет
DNS-записями для автоматической ротации IP-адресов.

## Архитектура

Мозг работает на сервере админки (рядом с backend и почтой) и состоит из пяти компонентов:

### Health Checker (опрос щитов) — в разработке

**Принцип работы:**
Раз в N секунд отправляет HTTP(S)-запрос на эндпоинт `/health` каждого щита.
Щит отвечает: «жив, нагрузка X, активных соединений Y».
Если три ответа подряд не получены — щит помечается как `offline`.
Отвечает на вопрос: «Щит технически жив или упал?»

**Отличие от Anomaly Detector:**
Health Checker проверяет сам сервер. Anomaly Detector проверяет доступность сервера из РФ.
Щит может быть жив (Health Checker: OK), но заблокирован РКН (Anomaly Detector: 🔴 БЛОКИРОВКА).

---

### Telemetry Collector (приём телеметрии от клиентов) — в разработке

**Принцип работы:**
HTTP-эндпоинт на сервере Мозга, куда клиентские приложения (V2Ray, Sing-box) периодически
отправляют отчёт: «Я клиент Z, подключён через щит X, скорость Y, ошибок W за последние N минут».
Данные агрегируются по щитам и сохраняются в `brain.telemetry`.

**Критично:**
IP-адреса клиентов не сохраняются. Только идентификатор клиента (хеш), идентификатор щита,
временная метка, метрики. Это питает Anomaly Detector данными без нарушения приватности.

---

### Anomaly Detector (поиск блокировок) — реализован

**Принцип работы:**
1. Для каждого активного щита берутся два последних замера телеметрии.
2. Сравнивается количество активных клиентов. Если падение >70% — подозрение.
3. Проверяется количество ошибок: рост в 5+ раз или абсолютное значение >20 — подтверждение.
4. Проверяется задержка: >500 мс — дополнительный признак блокировки.
5. Выносится вердикт: 🟢 Норма, 🟡 Подозрение, 🔴 Блокировка.

Пороговые значения настраиваются в коде (`DROP_THRESHOLD`, `ERROR_SPIKE`, `MIN_CLIENTS`).

**Отличие от Health Checker:**
Detector анализирует клиентскую телеметрию и отвечает на вопрос:
«Видят ли клиенты из РФ этот щит?»
Щит может быть технически жив, но недоступен из-за блокировки РКН.

---

### Action Dispatcher (реакция на блокировку) — в разработке

**Принцип работы:**
Получает событие «щит X заблокирован» от Anomaly Detector и выполняет цепочку:

1. **DNS Updater** — через Cloudns API убирает IP заблокированного щита из A-записи `vpn.ulysses.best`.
   Если есть резервный щит — добавляет его IP.
2. **Notification Sender** — отправляет уведомление администратору (email / Telegram).
3. **Recovery Planner** — ставит задачу в очередь: через 30 минут проверить заблокированный щит снова.
   Если разблокировали — возвращает в резерв (не в DNS, чтобы не дёргать клиентов).

**Важно:**
Action Dispatcher — единственный компонент, который вносит изменения.
Остальные только наблюдают и анализируют.

---

### DNS Manager (управление DNS через Cloudns API) — в разработке

**Принцип работы:**
Обёртка над API Cloudns. Умеет:
- Получать текущие A-записи домена `vpn.ulysses.best`.
- Добавлять IP в запись.
- Удалять IP из записи.
- Устанавливать TTL (всегда 60 секунд для быстрой ротации).

Используется Action Dispatcher-ом для автоматической смены IP и Admin API для ручного управления.

**Почему Cloudns:**
API быстрый, TTL поддерживается минимальный, не Cloudflare (приватность).

---

## Взаимодействие компонентов


Клиенты ──→ Telemetry Collector ──→ brain.telemetry
Щиты ──→ Health Checker ──→ brain.shields.last_health_check

Таймер ──→ Anomaly Detector ──→ читает brain.telemetry
│
▼ аномалия обнаружена
Action Dispatcher ──→ DNS Manager ──→ Cloudns API
│
▼
Notification Sender ──→ админ (email/Telegram)


## CLI-команды

```bash
# Заполнить базу тестовыми данными
uadmin brain seed

# Сгенерировать цикл телеметрии (норма)
uadmin brain telemetry

# Сымитировать блокировку щита
uadmin brain telemetry --anomaly --shield-id 1

# Проверить все щиты на аномалии
uadmin brain detect

# Состояние щитов
uadmin brain shield status

# Очистить все данные Мозга
uadmin brain reset

Хранение данных

Используется отдельная схема brain в общей PostgreSQL-базе. Таблицы:
Таблица	Назначение
brain.shields	Щиты: IP, страна, дата-центр, статус
brain.telemetry	Агрегированная телеметрия: клиенты, ошибки, задержка
brain.incidents	История блокировок: дата, действие, уведомление
brain.dns_state	Кэш DNS-записей, управляемых через Cloudns API
Интеграция

Мозг импортируется как CLI-модуль в общий uadmin (Click-группа).
Использует синхронный psycopg2 для запросов к БД, не зависит от FastAPI-приложения.
Конфигурация подключения — из .env (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASS).
